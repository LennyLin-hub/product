# 生产排程解决方案落地现状

本说明仅基于仓库当前实现，用于回答“已实现 / 部分实现 / 未实现”的落地情况与对应位置，方便后续对照补齐。

## 0. 总览

| 维度 | 当前状态 | 说明 |
| --- | --- | --- |
| 计划 | 已部分实现 | 已支持批次、工序任务、任务依赖 |
| 派工 | 已部分实现 | 已支持基于机台与班次日历的简化排程 |
| 执行 | 已部分实现 | 已支持任务事件日志与状态流转 |
| 异常 | 待完善 | 资源异常、动态重排、异常事件未完整闭环 |
| 完工 | 已部分实现 | 任务完工可驱动后续状态变化，但订单闭环仍待补齐 |

## 1. 背景痛点对照（落地现状）

### 1.1 效率低 → 已部分缓解
- 已实现：从“批次 → 工序任务 → 排程结果”形成基础自动化链路，减少人工拆解工作量。
  - 批次管理与释放：`product-pps`（生产批次 + 释放/撤销）
    - `product-pps/src/main/java/com/product/pps/service/impl/ProductionBatchServiceImpl.java`
  - 工序任务生成（3 工序）：`generateTask`/`retryGenerateTask`
    - `product-pps/src/main/java/com/product/pps/service/impl/OperationTaskServiceImpl.java`
  - 一键/全量排程：`/pps/assignment/schedule`、`/pps/assignment/scheduleAll`
    - `product-pps/src/main/java/com/product/pps/service/impl/TaskAssignmentServiceImpl.java`
- 已部分实现：排程入口已支持按参数切换最早开始、最早完工、交期优先三种策略。

### 1.2 准确率低 → 已部分缓解
- 已实现：排程时考虑“机台可用 + 班次日历 + 已排任务结束时间”，避免同机台时间重叠。
  - 机台资源可用性与班次约束：`chooseMachine`、`adjustToShiftStart`、`adjustForShiftEnd`
    - `product-pps/src/main/java/com/product/pps/service/impl/TaskAssignmentServiceImpl.java`
- 仍未实现/待完善：
  - 模具/人员/工位的协同占用约束（当前仅分配 machine_id）
  - 机模兼容性、能力矩阵、换型时间等硬约束
  - 计划产量与工艺参数联动（A2 产能公式未落地）

### 1.3 追溯性低 → 已部分缓解
- 已实现：任务事件日志（开始/暂停/恢复/完工），支持过程追溯。
  - 事件记录：`start/pause/resume/complete`
    - `product-execute/src/main/java/com/product/execute/service/impl/TaskEventServiceImpl.java`
  - 任务事件实体：`product-domain/src/main/java/com/product/domain/entity/TaskEvent.java`
- 仍未实现/待完善：
  - “计划-派工-执行-异常-完工”的完整事件流（当前无异常事件、无资源状态事件）
  - 完工后批次/订单自动闭环（状态联动）

### 1.4 成本高 → 已部分缓解
- 已实现：基本排程可避免同机台冲突，减少明显的资源闲置与重排成本。
- 仍未实现/待完善：
  - 资源利用率统计、瓶颈识别、冲突检测
  - 动态重排（异常/插单触发滚动重排）

## 2. 当前已落地的“计划 → 派工 → 执行”链路

### 2.1 计划侧：订单行拆批与批次管理
- 订单行数量预占用，拆批时做数量校验，避免超量拆批。
  - `product-pps/src/main/java/com/product/pps/mapper/OrderLineAllocationMapper.java`
  - `product-pps/src/main/java/com/product/pps/service/impl/ProductionBatchServiceImpl.java`
- 批次状态流转：PLANNED → RELEASED（执行前置约束）
  - `release`/`cancelRelease`
  - `product-pps/src/main/java/com/product/pps/service/impl/ProductionBatchServiceImpl.java`

### 2.2 计划到任务：工序任务自动生成
- 基于批次生成 3 个工序任务（SETUP/INJECT/POST_QC_PUTAWAY）。
- 任务的 `std_duration_min` 目前为“固定标准时长 × 批次数量”，未引入 A2 公式。
- 任务的 `earliest_start` 由“当前时间 + 前置工序累计时长”推算。
  - `product-pps/src/main/java/com/product/pps/service/impl/OperationTaskServiceImpl.java`
- 任务依赖关系已写入（SETUP → INJECT → POST）。
  - `TaskDependency` 写入逻辑同上文件

### 2.3 派工/排程：基于机台与班次日历的简化排程
- 排程逻辑：选择“最早可开始”的机台，写入 `task_assignment`。
- 约束包含：任务最早开始、机台已有排程结束时间、班次日历工作日/班次时间。
  - `product-pps/src/main/java/com/product/pps/service/impl/TaskAssignmentServiceImpl.java`
- 当前仅分配 `machine_id`，未考虑模具/人员/工位，也未计算换型时间。
- 当前自动排程算法实现要点：
  - 输入任务：仅对 `status=READY` 的工序任务进行排程（支持单任务与全量排程）。
  - 任务顺序：按 `batch_id + sequence` 顺序遍历（确保同批次工序先后）。
  - 资源候选：从 `resource` 表中筛选 `resource_type=MACHINE` 且 `status=AVAILABLE` 的机台。
  - 最早开始时间：取 `max(任务 earliest_start, 机台最后排程结束时间, 传入/当前时间)`。
  - 班次对齐：若早于班次开始则顺延至当班开始；若超过当班结束则顺延至下一工作日班次开始。
  - 排程结果：计算 `planned_start/planned_end`，写入 `task_assignment`，并将任务状态置为 `SCHEDULED`。

### 2.4 执行与追溯：任务事件日志
- 任务状态随事件更新：RUNNING/PAUSED/DONE。
- 事件记录绑定资源（当前取机台）。
  - `product-execute/src/main/java/com/product/execute/service/impl/TaskEventServiceImpl.java`

### 2.5 异步排程：任务记录、进度查询与超时兜底
- 已实现异步全量排程任务，调用后立即返回 `jobId`
  - `POST /pps/assignment/scheduleAllAsync`
- 已实现排程进度查询与分页查看
  - `GET /pps/assignment/scheduleJob/{jobId}`
  - `GET /pps/assignment/scheduleJob/list`
- 已实现超时兜底扫描，避免 PENDING / RUNNING 任务长期占用系统
  - `POST /pps/assignment/scheduleJob/sweepTimeout`
- 排程任务状态流转
  - `PENDING` → `RUNNING` → `SUCCESS` / `FAILED`
- 进度字段
  - `totalTaskCount`
  - `processedTaskCount`
  - `batchCount`
  - `progressPercent`
- 进度推送频率已与批次大小解耦，支持独立配置
  - `product.pps.schedule.progress-push-task-step`
  - 当前默认值：`50`
  - 说明：该配置只控制进度回调的触发频率，不影响分页大小和批量计算大小

### 2.6 进度编排：后续可拆分为独立服务
- 当前 `TaskSchedulingCoordinator` 已承接进度推送、耗时统计和全量排程编排，但仍然把“进度阈值判断”和“进度 DTO 组装”放在同一个类里。
- 如果后续继续拆分，建议抽出 `ScheduleProgressService`，专门负责：
  - 统一计算进度百分比
  - 控制进度推送节流策略（按任务数阈值、按时间间隔、按阶段切换）
  - 组装 `ScheduleProgressDTO`
  - 统一处理 `NO_TASK / STARTED / CALCULATING / SUCCESS / FAILED` 等阶段的进度语义
- 这样 `TaskSchedulingCoordinator` 只保留“何时调用进度服务”的编排职责，后续如果要改推送策略，可以只调整 `ScheduleProgressService`，不必改动排程主流程。

## 3. 解决方案三大目标对照（实现与缺口）

### 3.1 自动化排程
已实现（基础版）：
- 批次自动生成工序任务（3 工序）并写入预计时长。
- 一键/全量排程接口，自动选择可用机台并生成排程结果。

缺口：
- EDD/优先级驱动的队列排序与插单规则
- 机模兼容/能力矩阵/换型规则/A2 产能模型
- 多资源同步占用（机台 + 模具 + 人员/工位）
- 跨班次/跨日拆分与休息时段

### 3.2 资源优化
已实现（基础版）：
- 同机台排程顺序（`sequence_on_resource`）与最早可开始时间。
- 基于班次日历的可用时间窗口控制。

缺口：
- 资源状态事件（故障/保养/离班）
- 资源冲突检测（模具、人、工位）
- 利用率统计与瓶颈识别
- 动态重排触发机制

### 3.3 全流程追溯
已实现（基础版）：
- 任务事件日志与任务状态联动（开始/暂停/恢复/完工）。

缺口：
- 事件类型覆盖（派工、质检、异常、资源状态）
- 任务 → 批次 → 订单的完工闭环
- 实时进度看板（任务进度、良品/不良累计）
